## ①プロジェクトの背景、目的、目標

### 背景：操作を前提とするUIが作る構造的排除

日本の65歳以上人口は2025年に3,640万人を超え、そのうち約60%がスマートフォンを「使いこなせない」と報告している（総務省 令和5年通信利用動向調査）。入院患者・認知症患者・重度の運動障害者に至っては、既存のすべてのデジタル情報端末——スマートフォン・タブレット・スマートスピーカー——が機能しない。

問題の核心は「デジタルリテラシーの欠如」ではなく、**「操作を前提とした設計」そのもの**にある。スマートスピーカー（Alexa/Google Home）ですら「Hey Alexa」というウェイクワードの記憶と発話が必要であり、これは認知機能の低下した高齢者にとって深刻な障壁になる。認知症患者を対象としたAmbient Assisted Living研究（Pradhan et al., CHI 2018; Portet et al., INTERSPEECH 2013）は、音声インターフェース放棄の主因がウェイクワード発話の失敗であることを繰り返し報告している。高齢者の声は若年成人と比較して声門雑音・ジッタが増大し、既存ウェイクワードエンジンの偽陰性率は75歳以上で2〜5倍に達する（Vipperla et al., Speech Communication 2010）。

**問題の構造（具体的に）：**

| 利用困難の原因 | 具体的現れ方 | 現在の「解決策」の失敗点 |
|---|---|---|
| ウェイクワード依存 | フレーズを思い出せない/小声で検知されない | Siri/Alexa/Google — 同じ問題を持つ |
| 画面操作の困難 | タップ精度・認識速度が追いつかない | タブレット支援システム — 操作自体が障壁 |
| 情報提供のタイミング不一致 | 入院説明は退院前一回、患者の疑問は随時発生 | ICT看護システム — プッシュ型のみ |
| 記録の非対称性 | 患者の日常変化が見えない/記録コストが高い | 電子カルテ — 看護師記録負担が増大 |

これらの問題は個別の改善で解決できない。**操作を前提とするUI設計のパラダイムを変える**しかない。

### 本プロジェクトが開発するもの

Suzume-chanは**身体化AIターミナル（Embodied AI Terminal）**である——ぬいぐるみ型の物理デバイスと、施設内サーバ上の音声対話・適応・ログ生成システムを統合した、ハードウェアとソフトウェアの一体型プロダクト。

「Alexa in a plush toy」ではない。違いは次の4点に集約される：

1. **起動がない**：ウェイクワードや操作を必要とせず、持ち上げる・話しかけるだけで応答する（IMU+VAD複合トリガー。詳細は②参照）
2. **記憶が育つ**：会話のたびにユーザ固有の記憶・語り口・話題傾向を蓄積し、関係性が深まる（階層型適応システム。詳細は②参照）
3. **沈黙がない**：LLM推論中（1〜2秒）を振動触覚フィードバックで「聴いている」「考えている」に変換し、待機時間を相互作用に転換する（HRI '26で実証済み）
4. **看護師につながる**：会話から看護関連イベントを抽出し、スタッフ向けダッシュボードに提供する（双方向メディエーション）

### 目標（事業期間内）

**技術目標：**
- エンドツーエンド応答遅延：**2.5秒以内**（ユーザ発話終了→最初の音声出力、詳細は③参照）
- ウェイクワードゼロの起動設計（IMU + Silero VAD + 近接ゲート）の実装・検証
- 階層型適応システム（Profile + RAG + 周期的LoRAバッチ更新）のv1実装
- RP2040 + RPi Zero 2W構成のデバイスv2プロトタイプ完成（BOM目標¥10,000以下）
- 施設設置型サーバ（Mac mini M4 Pro）との分散推論パイプラインの完成

**社会実装目標：**
- 高齢者施設での実証実験（2施設以上、各2週間以上）
- 医療倫理審査承認・プライバシー設計の完了
- 量産プロトタイプv2の設計・製造パートナーとの原価合意

### 未踏軸1：低エージェンシーユーザ向けウェイクワード不要起動アーキテクチャ

既存の音声インターフェースはすべてユーザの能動的な起動行為（コマンド発話・ボタン押下・タップ）を前提とする。認知症・高齢者・重症患者では、この前提が成立しない。本プロジェクトは**ウェイクワードを排除した複合トリガーアーキテクチャ**を設計・実装する。

**3ステージ起動設計（ゲーティング構造）：**

```
Stage 1: 近接/存在ゲート（消費電力: ~10 mW）
  → BLE RSSI変化 + PIRモーションセンサによるユーザ存在検知
  → このゲートが開いていない間はStage 2,3は起動しない

Stage 2: 物理インタラクショントリガー（消費電力: ~25 mW）
  → IMU（LIS2DH12）: ピックアップ検知（デバイスが持ち上げられた）
  → 静電容量タッチ（ぬいぐるみ表面に導電布）: 握り込み検知
  → このトリガーで音声受付モードを起動（LED点灯）

Stage 3: 音声活動検知（消費電力: ~45 mW）
  → Silero VAD (RP2040上 TFLite-Micro版: ~3 ms/chunk, 30 ms窓)
  → マイクアレイ（2ch）によるビームフォーミングで正面方向に感度集中
  → 発話終端検知後、音声をOpus圧縮してWiFi経由でサーバに送信
```

**既存ウェイクワードアーキテクチャとの比較：**

| 特性 | ウェイクワード方式 | 本提案（複合トリガー） |
|---|---|---|
| 高齢者偽陰性率 | 高（2〜5倍） | 低（物理トリガーは声量非依存） |
| 施設内誤起動 | 高（TV・スタッフ会話） | 低（近接ゲート + ビームフォーミング） |
| 認知負荷 | 高（フレーズ記憶が必要） | ゼロ（持ち上げる/話しかけるだけ） |
| プライバシー（常時録音） | あり | なし（Stage 1,2通過時のみ録音） |

**フォールバック**：ボタン（スタッフ操作用）をデバイス底部に設置。完全に発話困難なユーザへの対応および施設スタッフによる代理操作を可能にする。

**物理ミュートスイッチ**：LEDインジケータ付き可視ミュートスイッチ。臨床プライバシー要件（医療情報安全管理ガイドライン）への対応。

この起動アーキテクチャの設計・実装・評価（偽陰性・偽陽性率の定量測定、施設環境での動作検証）を本事業期間の主要な技術的貢献の一つとする。

### 未踏軸2：破滅的忘却を防ぐ階層型適応システム

「パーソナルRAG」のような単純な記憶機構ではなく、**安全性を担保しながらユーザへの適応を段階的に深める4層アーキテクチャ**を設計・実装する。

**なぜオンラインLoRAではないか：** 真のオンラインLoRA（推論中のリアルタイム重み更新）は推論の2〜4倍のVRAMを要求し、勾配ステップの不安定性が臨床環境では許容できないリスクになる。また、会話ログを学習データとして使用するには倫理審査承認が前提となり、事業期間内にリアルタイム更新を実現するには規制面の障壁が大きい。

**4層適応アーキテクチャ（下層から上層へ）：**

**Layer 0: 固定ベースLLM**（更新しない）
- Llama 3.2-14B Q4_K_M（~8GB）、基本的な日本語対話能力・安全フィルタを保持
- 基盤モデルは事業期間中更新しない。安全性・一般能力の劣化リスクをゼロにする

**Layer 1: ユーザプロファイル注入**（毎セッション更新）
- 構造化プロファイル（名前・家族構成・服薬・好みの話題・コミュニケーションペース）をシステムプロンプトとして注入
- 最大800トークン（TTFT増加コスト: +100〜150ms）
- 会話後に自動抽出・更新。看護師・介護士がダッシュボードから確認・修正可能
- 実装コスト: SQLite + 抽出用NLPパイプライン、2〜3週間

**Layer 2: エピソード記憶RAG**（セッション間で蓄積）
- 会話サマリをall-MiniLM-L6-v2（22MB）でエンコードしFAISSインデックスへ格納
- 検索レイテンシ: ~30ms（1,000エントリ規模）
- 会話開始時にトップ5エピソードを自動注入（「先週娘さんが来訪した」等）
- **Memory Importance Score**: 参照頻度×経過時間×感情強度の積で重み付け。頻出・重要な記憶を優先的に注入

**Layer 3: 行動ポリシーモジュール**（セッション内で動的更新）

観測変数（入力）:
- 発話速度（WPM）: ASR出力のトークン数÷発話時間で算出
- ポーズ長（ms）: ターン間の無音期間
- 繰り返し質問率: 前3ターン内の同義質問出現率（埋め込み類似度で検出）
- 応答遅延（ユーザ側）: Suzume-chanの発話終了からユーザ発話開始までの時間

制御変数（出力）:
- TTS速度: 基準速度±20%の範囲で調整（Style-Bert-VITS2のspeedパラメータ）
- LLMへの生成長ヒント: システムプロンプトに「1〜2文で答えよ」「詳しく説明せよ」を動的追加
- プロアクティブ発話トリガー: ユーザが30秒以上沈黙した場合、前回の話題に基づく先行発話

制御則（初期実装）: 閾値ベースのif-thenルール（例：WPM<60かつポーズ>3秒 → TTS速度を15%低下かつ文長を50%削減）。評価フェーズでルールを施設実験データから最適化する。より高度な実装として軽量強化学習（Contextual Bandit、ユーザの「ありがとう/もう一度言って」シグナルを報酬）をフェーズ2での研究候補として保留する。

**Layer 4: 周期的LoRAバッチ適応**（週次更新、研究フェーズ）
- Layer 0〜3で対応できない長期的な語り口・説明スタイルの変化をLoRAで吸収
- 会話ログ（倫理審査承認・患者同意取得後）からRank-8 LoRAを週次バッチ学習
- Mac mini M4 Pro上でmlx-lmを使用：14B Q4モデルへのLoRA学習、100〜500会話サンプルで30〜90分
- **破滅的忘却対策**: 学習データに汎用日本語対話サンプル（10〜20%）を混入（リプレイバッファ）
- **安全フィルタ**: 生成テキストを既存安全フィルタと比較し品質劣化を検出した場合は直前バージョンにロールバック
- **スコープ明確化**: Layer 4はこの事業期間における研究貢献として位置付ける。事業期間内での実証にとどめ、「製品としての自律学習」は次フェーズの開発課題とする

この4層設計は「オンライン学習の不安定性」と「静的プロンプトの限界」の間を、エンジニアリング的に合理的な方法で橋渡しする。

### 未踏軸3：振動触覚による待機時間の相互作用化（Embodiment as Latency Mediator）

HRI '26実証（27名・27セグメント定性分析）の主要知見：**振動触覚フィードバックがない場合、LLM推論中の1〜2秒の沈黙は「無反応」または「故障」として知覚され、ユーザは同じ質問を繰り返す。**振動があると同じ遅延が「考えている」として自然に受容される。

この差異はUX上の話ではなく、**システム負荷と情報品質に直結する機能的問題**である。繰り返し質問は（a）ASR再処理のコスト増、（b）LLMコンテキスト汚染、（c）ユーザの混乱・不信感の増大を引き起こす。

**3段階振動パターン（DRV2605L LoRA触覚アクチュエータ）：**

| フェーズ | パターン | 目的 |
|---|---|---|
| Listening（発話受付中） | 0.5Hz穏やかなパルス（30ms on / 1970ms off） | 「聴こえているよ」の継続確認 |
| Thinking（推論中） | 2Hz連続振動（弱強度, 70/255） | 「考えているよ」の能動的表示 |
| Speaking（応答直前） | 短いバースト確認（3×短パルス） | 「話すよ」の予告、ユーザの注目準備 |

この振動パターンの知覚効果を、今回の事業期間で音声のみ条件と振動あり条件の被験者間比較実験（30名以上）で定量評価し、学術的に確立する（CHI/HRI投稿目標）。

### 未踏軸4：施設展開可能な分散推論アーキテクチャ

「完全ローカル」を硬直的に主張するのではなく、**配備環境に応じた柔軟な計算分散設計**を提供する。

```
[施設内展開モデル / On-Premise]
  ┌────────────────────────────────────┐
  │ Plush Device (RP2040 + Zero 2W)   │
  │  ● 常時待機: ~30 mW              │
  │  ● Stage1-3 起動ロジック          │
  │  ● Opus音声ストリーミング送信     │
  │  ● 振動触覚制御 (DRV2605L)       │
  │  ● キャッシュTTS再生(fallback)    │
  └──────────────┬─────────────────────┘
                 │ WiFi 802.11n (LAN内, RTT<15ms)
  ┌──────────────▼─────────────────────┐
  │ 施設サーバ (Mac mini M4 Pro 48GB) │
  │  ● distil-Whisper ASR: 200-400ms  │
  │  ● 14B Q4 LLM (llama.cpp Metal): │
  │      TTFT ~1,000-2,000ms          │
  │  ● FAISS RAG検索: ~30ms           │
  │  ● Style-Bert-VITS2 TTS:          │
  │      ストリーミング, 400-700ms    │
  │  ● Layer 3 ポリシーモジュール     │
  │  ● ログ生成/ダッシュボード        │
  │  ● 週次LoRAバッチ学習ジョブ       │
  └────────────────────────────────────┘

  目標RTT（発話終了→最初の音声）: 2.0〜2.5秒
  同時接続: 最大8台（時分割処理, 検証予定）

[在宅展開モデル / Hybrid Cloud]（将来フェーズ）
  Plush Device → WiFi → 軽量クラウドAPI
  (家庭向けプライバシー設定に応じてon/off可能)

[ネットワーク断絶時のFallback]
  Device上SDカード(8GB)にキャッシュ済みTTS音声を格納
  「少々お待ちください」「確認してから答えます」等の
  定型応答25種をオフライン再生可能
```

**サーバ管理の現実性：** Mac mini（Mac mini M4 Pro相当、実売約15万円）は施設IT部門が管理可能な製品カテゴリであり、特別なデータセンター設備は不要。10台のデバイスを1台のサーバでサポートする設計は、施設当たり15万円の追加投資（デバイス10台×3万円=30万円との合計45万円）で成立する。

### 未踏性の総括：なぜこれは「ぬいぐるみに入れたAlexa」ではないか

| Alexa / Google Home | Suzume-chanターミナル |
|---|---|
| ウェイクワード必須 | IMU+VAD複合トリガー（ウェイクワード不要） |
| クラウド完全依存 | 施設内オンプレミス推論（医療プライバシー対応） |
| セッション記憶なし | 4層適応（プロファイル+RAG+ポリシー+LoRA） |
| 身体フィードバックなし | 振動触覚による待機時間の相互作用化 |
| 単方向（ユーザ→クラウド） | 双方向（患者→情報+会話→看護ログ） |
| 汎用エンターテイメント設計 | 低エージェンシーユーザ向け臨床設計 |

**ぬいぐるみ形態の選択根拠（補足）：** 形態の受容性は施設・文化・個人によって異なることは認識しており、実証フェーズの調査項目に含める（施設スタッフ・入居者へのアンケート）。既存研究（Paro: 動物型ロボットの施設受容性）と本チームのWISS 2025での観察（参加者が自然にぬいぐるみに話しかけた事例）から、柔らかく生物的な形態はTech嫌いのユーザの心理的障壁を下げることを支持する証拠がある。施設連携打診先2施設からの担当者フィードバックでも形態への明示的な拒否反応はなかった（2026年3月現在）。

## ③開発の具体的な進め方

### 現状（実証済み事項）

**HRI '26論文（査読付き採録、2026年3月）で実証済み：**
- 手のひらサイズぬいぐるみデバイス（マイク・スピーカー内蔵、Bluetooth）
- ASR（Apple Speech framework）→LLM（gpt-oss-120B）→RAG（LangChain+ChromaDB）→TTS（Style-Bert-VITS2）の完全ローカルパイプライン動作
- 振動触覚フィードバックプロトタイプの設計・デモ
- 27名参加者によるワークショップ評価（定性分析）

**WISS 2025発表（2025年12月）で実証済み：**
- 研究者が自身の研究をSuzume-chanに「教え込み」、来場者がそれを質問して対話するシステムのライブデモ
- Embodied Information Hubコンセプトの公開評価

**現プロトタイプの明確な限界：**
- Mac Studio（128GB UMA）依存、施設展開不可
- ウェイクワード（Apple Speech）依存、高齢者での失敗が多い
- 振動触覚は試作レベル（定量評価なし）
- 個人記憶はセッション内のみ（長期記憶なし）
- 同時接続は1台のみ

| **BOM合計（単品）** | | | **≈ ¥8,850** |

**目標小売価格：¥30,000〜¥50,000**（3〜5x BOM倍率、B2B施設向けには¥30,000台を想定）

**RP2040 + RPi Zero 2W構成を選択した理由：**
- ESP32-S3単体：コスト最安（BOM ¥6,000〜）だがRAM 8MBの制限でOpus+WiFi同時処理のマージンが薄く、v1では安全策をとる
- RPi 4B単体：能力過剰・コスト過大（BOM ¥15,000〜）・電力5W+でバッテリー駆動に不利
- **RP2040+Zero 2W**：常時ONのRP2040は~30mW待機、Zero 2WはLinux環境でOpusストリーミング・OTA更新・高度なVAD処理を担当。コスト・能力・電力のバランスが最良

**施設サーバ：**
- Mac mini M4 Pro 48GBユニファイドメモリ（実売約15万円）
- 同時接続8台対応（LLM時分割処理、1台あたりのレイテンシ影響: +200〜400ms）
- 将来的にJetson AGX Orin 64GB（約30万円）への移行でさらに高い同時接続数に対応

```
ユーザ発話終了（VAD終端検出）: +300〜600ms
Audio → Opus圧縮 → WiFi送信: +20〜50ms
distil-Whisper ASR (5秒発話): +200〜400ms
FAISS RAG検索: +30ms
プロファイル注入 + LLM prefill: +500〜1,200ms
LLM 1文生成(〜20トークン): +300〜500ms
Style-Bert-VITS2 TTS 第1文: +400〜700ms
WiFi返送 + バッファ: +50〜100ms
─────────────────────────────────────────
合計（first audio）: 1.8〜3.6秒（平均 約2.5秒）
```

**最適化施策（フェーズ1で実装）：**
1. ストリーミングTTS：LLM出力の第1文節（10〜15トークン）でTTS開始（体感レイテンシ-400ms）
2. 投機的プリフィル：VAD終端検出を見越してRAG検索を先行実行（-100〜200ms）
3. コンテキスト積極的刈り込み：プロファイル+RAGコンテキスト上限600トークン（prefill時間安定化）
4. Layer 3ポリシー：高速モード（7B Q4、~70tok/s）/標準モード（14B Q4）をユーザ応答速度に応じて切替

### 克服すべき課題と解決策

**リスク1（高）：LoRAバッチ更新のための倫理審査・同意設計**
会話ログを学習データとして使用するには患者・施設・倫理委員会の同意が必要。解決策：Layer 4は研究フェーズとして位置付け、同意取得した参加者のみに限定適用。同意書設計と倫理申請を7〜8月に完了させる（小関が専門担当）。Layer 4が事業期間内に実現できない場合も、Layer 1〜3で主要な価値提供は成立する。

**リスク2（高）：施設実証スケジュール**
施設の意思決定プロセス（理事会→倫理委員会→スタッフ合意）に2〜3ヶ月かかる。解決策：7月時点（事業開始月）に既存打診済み2施設への正式依頼を即時開始。並行して大学院生・多様な年齢層での予備実験（10〜15名）を10月に実施し、施設実証の遅延に備える。

**リスク3（中）：RP2040でのSilero VAD実行可能性**
RP2040のSRAMは264KB。Silero VAD TFLite-Micro版は重みが~100KBだが、実行時スタック・テンソルアリーナ含めて200KB超を使用し、264KBに収まらない可能性がある。解決策の優先順位：（a）RP2040でのSilero VAD Tiny（INT8量子化済み、~60KB重み）の動作確認を事業開始直後に実施。不可能な場合は（b）RP2040をIMU+LED+タッチ制御のみに限定し、VAD処理をZero 2W（Linux、計算資源十分）に移管する。（b）の場合、Zero 2Wの常時電力が~300mWになるが、有線給電（USBアダプタ）との組み合わせで問題なし（バッテリー運用は補助的に）。いずれの構成でも3段階起動の設計原則は維持できる。

**リスク4（中）：看護ログ分類器の訓練データ**
施設固有の会話から医療関連イベントを分類するBERTモデルの訓練データ調達は非自明。解決策：MedTxt（東京大学公開医療テキストコーパス）・NTCIR医療NERデータセットを起点として活用する。施設アノテーションは倫理審査承認後にフェーズ2（10〜12月）で実施する。事業期間の最初の段階はGiNZA+規則ベースNERで十分機能する（「○○が痛い」「薬を飲んだ」等の表面的パターンマッチで大部分を捕捉可能）。BERTファインチューニングはフェーズ3の研究成果として位置付ける。

**リスク5（中）：LoRAホットスワップの技術的実現可能性**
llama.cppのLoRAアダプタのダイナミックローディング（モデル再ロードなし）は2025年以降の主要バージョンでサポートされている（`llama_model_apply_lora_from_file` API）。村上が事業開始前に動作確認を実施する。問題がある場合は、セッション間のアダプタ切り替えをモデルの高速再ロード（~5秒）で代替し、セッション開始前の1回だけ実行する設計で対応する（実用上の影響は最小限）。

**リスク6（中）：施設ネットワーク対応**
多くの介護施設はWi-Fiの整備が不十分。解決策：施設への導入提案に「Wi-Fi APの設置（1〜2台、費用¥2〜3万円）」を含めたパッケージとして設計し、初期導入支援に組み込む。

**優先度の明確化（スコープ管理）：** 8ヶ月で最も重要な技術的成果物は、優先度順に：①起動アーキテクチャ（IMU+VAD複合トリガー）の施設環境での動作実証、②Layer 1〜3適応システムの完成と評価、③施設実証実験の実施とデータ収集、④Layer 4 LoRAバッチパイプラインの実装（できれば）。④は倫理審査・技術的実現可能性に依存するためオプション扱いとし、①〜③を確実に達成することを最優先とする。

本未踏プロジェクトの開発内容（量産プロトタイプ設計・施設実証・事業化計画・製造パートナー連携）は大学における学術研究の範囲を超えており、大学が同テーマで商業的事業を並行して行っているわけではない。

筑波大学サイバー医学研究センターの若手研究者支援助成は本プロジェクトと研究テーマが重複するが、当該助成は論文発表・学術成果を対象とするものであり、本未踏提案の目的（量産・事業化・施設実装）とは明確に異なる。所属組織（筑波大学）からの了解は取得済み（form1参照）。
