## ① プロジェクトの背景、目的、目標

### 背景①：独居の高齢者が「一人でいる」問題

日本の独居高齢者は約750万人（2020年、国勢調査）。そのうち軽度認知障害（MCI）を持つ人は約400万人と推計される。子や孫が離れて暮らすこの層にとって、日常の「誰かに話しかける」行為はほぼ失われており、孤独感・認知機能低下・情報断絶が三位一体で進行する。

問題の核心は**「話し相手がいない」**ことである。介護度が低く施設に入らない段階の人々（要介護0〜1相当）は支援の隙間に落ちており、既存のデジタルデバイスも「操作できる人」を前提としているためアクセスできない。

### 背景②：家族が「知ることができない」問題

離れて暮らす家族は、親の「今日どうだったか」を知る手段を持たない。電話は相手が出なければ終わり、スマホは親が使えない。施設に入るほどではないが、状態の変化を見逃したくない——この需要を満たすプロダクトは現在存在しない。

### 背景③：スマートスピーカーが解かない理由

「Alexaがあるのでは」という反論がある。しかしスマートスピーカーはこの問題を解かない。**ウェイクワード（「Hey Alexa」）を思い出し、声量を出して発話できる人**を前提とする設計であり、75歳以上での偽陰性率は若年層の2〜5倍に達する（Vipperla et al., 2010）。さらにスマートスピーカーには記憶がなく、関係性がなく、身体がない。毎回ゼロから話しかける「道具」である。

```
【図1：なぜ既存ソリューションが問題を解かないか】

  独居高齢者の問題        既存デバイスの限界
  ┌────────────────┐   ┌──────────────────────────────────────┐
  │ 話し相手がいない  │──▶│ スマートスピーカー：ウェイクワード失敗多数│
  │ 家族と繋がれない  │──▶│ スマートフォン：操作自体が障壁          │
  │ 状態が記録されない│──▶│ 見守りセンサー：会話・記憶機能なし      │
  │ 認知機能が低下する│──▶│ タブレット支援：タッチ操作が困難        │
  └────────────────┘   └──────────────────────────────────────┘
          ↓
    「操作しなくていい、そこにいてくれる存在」が必要
```

### 目的：「そこにいる存在」としての身体化AIエージェント

**Suzume-chanは、独居高齢者の部屋に置くと自然に「一緒にいる」ぬいぐるみ型AIエージェントである。**

声をかければ応じる。話しかけなくても振動でそこにいることを伝える。毎日少しずつ話し方・好み・思い出を覚えていく。ペットや古い友人のように**関係が育つ**。この「いる感（Social Presence）」と「タンジブルな身体性」こそが本プロジェクトの本質であり、スマートスピーカーやロボットとの根本的な差異である。

家族のスマートフォンには毎晩「今日のサマリー」が届く——今日何を話したか、声のトーンはどうだったか、気になるキーワードはあったか。「お母さんが元気かどうか」が手軽にわかる唯一の方法になる。

## ② 開発に関する未踏性の主張、期待される効果

### 未踏性の核：「話し方は覚えるが、1年前の朝ごはんは覚えない」3層記憶アーキテクチャ

本プロジェクトの技術的中核は、**ヒトの記憶構造を模した3層オンライン適応システム**である。既存の会話AIはRAGによる検索か静的プロンプト注入に留まり、「何を長く覚えるべきか」「何を忘れるべきか」を自動的に判断する機構を持たない。本提案は記憶の重要度・時間減衰・再参照頻度から適切な記憶層への**自動ルーティング**を実装する。

```
【図2：3層記憶アーキテクチャとルーティング】

  ┌──────────────────────────────────────────────────────┐
  │  会話入力 → ルーティングエンジン（重要度×時間×頻度）   │
  └──────────────┬──────────────┬──────────────────────┘
                 │              │                      │
       ▼短期記憶              ▼中期記憶            ▼長期記憶
  ┌──────────┐       ┌──────────────┐    ┌────────────────┐
  │Context   │       │ RAG          │    │ LoRA Adapter   │
  │Window    │       │ (FAISS)      │    │ (週次バッチ)    │
  │~10ターン │       │ ~1,000エントリ│    │ Rank-8         │
  │セッション内│       │ 数日〜数ヶ月  │    │ 数ヶ月〜永続    │
  └──────────┘       └──────────────┘    └────────────────┘
  「今何の話?」      「先週娘が来た」      「ゆっくり話す」
  「さっき言った薬」  「病院の予約3/25」   「鳥の話題が好き」
                     「最近眠れていない」  「敬語より友達口調」
                                          ↑話し方・接し方・
                                           価値観はLoRAで学習
  ───────────────────────────────────────────────────────
  例：1年前の朝ごはん → 短期記憶から自然に消去（重要度スコア低）
     娘の名前・趣味 → 長期記憶に昇格（参照頻度高・感情強度高）
```

**ルーティング判定ロジック（Memory Importance Score）：**

```
MIS = 参照頻度(f) × 感情強度(e) × 時間近接性(1/t) × 話題カテゴリ重み(w)

- MIS > θ_long  → LoRA学習キューに追加（週次バッチ）
- θ_mid < MIS ≤ θ_long → RAGインデックスに追加・更新
- MIS ≤ θ_mid  → Context Windowのみ（セッション後に破棄）
```

**なぜ3層か（既存手法の限界）：**

| 手法 | 問題 |
|------|------|
| 静的プロンプトのみ | 個人適応が浅い。セッションを跨ぐ記憶がない |
| RAGのみ | 膨大な過去ログが等価に検索される。「何を重視するか」を学習しない |
| オンラインLoRAのみ | VRAM2〜4倍・勾配不安定。臨床環境では許容不可 |
| **本提案：3層自動ルーティング** | 重要な記憶はLoRAで人格化、中期的な出来事はRAGで検索、直近はContext Window。計算コスト・安定性・適応深度の最適バランス |

### 未踏軸①：「いる感」を生む ウェイクワード不要・身体化トリガー起動

```
【図3：3ステージ起動アーキテクチャ（認知負荷ゼロ）】

  Stage 1: 近接ゲート (~10 mW 常時待機)
    BLE RSSI変化 + PIRモーション → ユーザ存在検知
    ※人がいない間はマイクを起動しない（プライバシー・省電力）

  Stage 2: 物理トリガー (~25 mW)
    IMU: ぬいぐるみを持ち上げた ──┐
    導電布タッチ: 触れた・握った  ──┤→ 応答モード起動（LEDが光る）
                                   │  「持ち上げれば話せる」
  Stage 3: 音声活動検知 (~45 mW) ←┘
    Silero VAD on RP2040 (~3ms/chunk)
    → 発話終端検知 → Opus圧縮 → WiFi送信

  【高齢者偽陰性率の比較】
  ウェイクワード方式：75歳以上で若年層の2〜5倍 → 本提案：物理トリガーで解消
```

また、LLM推論中（1〜2秒）は**振動触覚フィードバック**により「聴いているよ」「考えているよ」を伝達する。HRI '26実証（27名）で確認：振動なし条件では推論中の沈黙が「壊れた」と知覚されて繰り返し発話が多発したが、振動あり条件では自然に待機された。この「待てる感覚」こそが継続使用の鍵である。

### 未踏軸②：API優先アーキテクチャと将来ローカル移行の二段設計

**プロダクト開発はAPIサーバー前提で開始する。**推論バックエンドを自前サーバーに縛らず、OpenAI API・Claude API・OSS APIサーバー（llama.cpp server / vLLM）をインターフェース共通化した抽象層の上に構築する。これにより初期開発速度・品質・コストのバランスを最適化しながら、研究サイドではローカルLLMを継続して将来の施設展開・プライバシー対応を研究する。

```
【図4：システム全体アーキテクチャ（API優先・ローカル研究の二段構成）】

  [ぬいぐるみデバイス]  RP2040 + RPi Zero 2W  ~30mW待機
   ・3ステージ起動ロジック
   ・Opus音声ストリーミング
   ・振動触覚制御 (DRV2605L)
        │ WiFi（家庭LAN / 施設LAN）
        ▼
  [バックエンドサーバ（抽象化APIレイヤー）]
   ・ASR:   distil-Whisper or Whisper API
   ・LLM:   OpenAI API / Claude API / OSS API サーバー（切替可能）
   ・3層記憶ルーティングエンジン（本提案の技術核）
   ・RAG:   FAISS / ChromaDB
   ・TTS:   Style-Bert-VITS2 / VOICEVOX / クラウドTTS
   ・週次LoRAバッチ学習ジョブ
   ・家族向けサマリー生成パイプライン

  ┌──────────────────────┬──────────────────────────────┐
  │ プロダクト開発ルート  │ 研究ルート（並行）            │
  │ API優先               │ 完全ローカル（施設向け）      │
  │ 高速開発・高品質推論  │ ローカルLLM (llama.cpp)       │
  │ スケール容易          │ 医療プライバシー規制対応      │
  │ 初期コスト最小化      │ 施設展開・オフライン動作研究  │
  └──────────────────────┴──────────────────────────────┘

  目標RTT（発話終了→最初の音声出力）: 2.0–2.5秒（API利用時）
```

**将来移行パス**：在宅向けプロダクトがスケールした後、施設展開・医療情報規制対応フェーズでオンプレミスローカル推論に段階移行する。研究サイドでのローカルLLM実証が、その移行の技術的根拠となる。

### 期待される効果

- **独居高齢者**：毎日「話せる存在」による孤独感軽減・認知機能低下の抑制（会話継続による認知刺激）
- **家族**：毎日のサマリーによる遠隔見守り。「今日お母さんは元気だった」が手軽にわかる
- **介護・医療**：記録業務の自動化。会話ログからの状態変化検知（将来研究フェーズ）
- **社会**：スマホを持てない人口層（約1,000万人規模）の初の生活ログデータ基盤

## ⑤ 事業化・社会実装の新規性・優位性、想定するターゲットと規模

### ターゲット：独居高齢者とその家族（CtoC）

**一次ターゲット（購買者）**：離れて暮らす子・孫世代（30〜50代）。「親に使わせる」のではなく「**親のそばに置く**」プロダクト。「お母さんのそばにSuzume-chanを置いてあげる」という購買行動。

**エンドユーザ**：軽度認知障害または軽度身体障害を持つ独居高齢者（65〜85歳、要介護0〜1相当）。スマホが使えない、電話に出られないことが多い層。

**二次ターゲット（将来）**：高齢者施設（有料老人ホーム・特養）。在宅での実証実績を持って施設展開。

### カテゴリ定義：「身体化AIコンパニオン」は既存市場に存在しない

| 競合 | 問題 |
|------|------|
| Alexa / Google Home | ウェイクワード必須・記憶なし・施設医療規制に非対応 |
| パロ（AIST）| 感情接触特化・受動デバイス。情報・記憶・ログ機能なし |
| ペッパー（SoftBank）| 月額¥100万超・UIあり・家庭利用不可 |
| 見守りセンサー | 状態検知のみ。会話・関係形成・記憶機能なし |
| **Suzume-chan** | ウェイクワード不要・3層記憶・身体性・家族連携・将来ローカル移行可 |

初期はAPI優先で開発速度と品質を両立し、施設展開フェーズでのオンプレミス移行を研究サイドで並行準備する。「ローカル移行できる設計」がプライバシー規制における護城河として機能し、大企業（クラウド依存構造）が追随しにくい差別化要因となる。

### 新規性の核：スマホ非利用者の生活ログデータ基盤

スマートフォンは1日数百件のセンサーデータ・行動ログを生成し、GoogleやAppleの巨大データビジネスを支える。しかし**スマホを使えない高齢者のデータは存在しない**。Suzume-chanは、これまでデジタルの痕跡を残さなかった約1,000万人（独居高齢者・軽度MCI層）の日常生活・会話・感情の記録を唯一生成できるデバイスである。

## ⑥ 事業化・社会実装の具体的な進め方

### ビジネスモデル：CtoC × データ × 施設への段階的展開

**基本構造（ハード × SaaS × データ）：**

| 収益源 | 内容 | 単価 |
|---|---|:---:|
| デバイス販売 | 家族が購入してプレゼント。Amazon/ECサイト経由 | ¥30,000/台 |
| 月額メモリーサービス | クラウドバックアップ・家族サマリー配信・ソフト更新 | ¥1,500/月 |
| 施設向けパッケージ | デバイス一括導入＋サーバ設置＋スタッフ研修 | ¥500,000〜/施設 |
| データビジネス（将来）| 匿名化された高齢者生活ログの研究・ヘルスケア提供（opt-in）| — |

**データビジネスの構造的優位性：**

スマホを持てない高齢者の生活データは現在どこにも存在しない。Suzume-chanが家庭に普及するほど、**他のどの企業も持っていない「スマホ非利用者の日常ログ」が蓄積する**。このデータは製薬会社・保険会社・医療研究機関・介護事業者にとって高い価値を持ち、opt-in同意を前提とした匿名化データの提供が将来の収益柱になる。

**施設展開への橋渡し：**

CtoC普及→在宅実証実績→「Suzume-chanを知っている入居者」が施設に入る→施設側から導入需要が発生、という自然な流入経路を設計する。施設は個別家族向けデバイスの「拡張版」として受け入れやすく、販売交渉のコストが下がる。

### ゴートゥーマーケット戦略

| フェーズ | 時期 | チャネル | KPI |
|---|---|---|---|
| Phase 0: プロダクト検証 | 事業期間内 | 在宅実証10名・家族ヒアリング | SUS・継続使用率・サマリー有用性評価 |
| Phase 1: 初期販売（B2C）| 2027年前半 | Amazon・EC・看護師口コミ | 500台稼働・月額継続率 >70% |
| Phase 2: チャネル多様化 | 2027年後半 | 介護用品店・薬局・訪問看護ST | 3,000台稼働・施設引き合い5件以上 |
| Phase 3: 施設展開 | 2028〜 | 在宅実績を軸に施設直販 | 施設50件・在宅5,000台 |

**「家族がプレゼントする」購買文脈の設計：**
「父の日・敬老の日・誕生日ギフト」として、子や孫がECで購入して実家に送る。初期設定はスマートフォンアプリから遠隔で行えるよう設計（家族がセットアップ、高齢者はただ置くだけ）。
