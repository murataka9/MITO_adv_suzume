## 2. 採択した設計と棄却した代替案

### 2.1 起動/アクティベーション設計

#### 採択案：3段階複合トリガー（IMU + VAD + 近接ゲート）

**設計根拠：**
- Stage 1（近接ゲート）：BLE RSSI + PIR。ユーザが近くにいないときはVADすら起動しない。消費電力削減と施設内誤起動防止の両立
- Stage 2（物理トリガー）：IMUピックアップ検知（LIS2DH12: threshold 1.5g, duration 5ms）+ 静電容量タッチ（Velostat）。この2つはウェイクワードと異なり声量・発音能力と完全に独立
- Stage 3（VAD）：Silero VAD TFLite-MicroをRP2040で実行。30ms窓、1フレーム~3ms処理。偽陽性対策にビームフォーミング（2マイク）を組み合わせる

**物理ミュートスイッチ**：デバイス側面に配置。LED（赤：ミュート中、緑：受付中）で状態を可視化。厚労省医療情報安全管理ガイドラインの要求する「収集されているデータの明示」に対応。

#### 棄却案1：ウェイクワード（「すずめちゃん」等）
**棄却理由：**
- 高齢者・認知症患者の偽陰性率が2〜5倍（文献的に確認済み）
- フレーズの記憶自体が認知負荷
- 施設内でのTV・スタッフ会話による誤起動リスク
- Porcupineなどの商用エンジンでも高齢者向けに追加訓練が必要（期間内に実現困難）

#### 棄却案2：常時ストリーミング（Continuous Listening、トリガーなし）
**棄却理由：**
- プライバシー懸念が医療施設では特に深刻（職員・他患者の会話が記録される）
- 電力消費：Zero 2WでのWiFiストリーミング常時稼働は電池駆動と相性が悪い
- 施設内でのネットワーク帯域への影響
**部分採用：** 近接ゲートが開いているときはVADをストリーミングし、音声活動を検知した瞬間から送信開始する「VAD-gated continuous listening」として折衷的に実装

#### 棄却案3：カメラベース顔検知（Proximity/attention detection）
**棄却理由：**
- 医療施設での撮像は個人情報・プライバシー問題が深刻
- 患者の同意取得コストが高い
- センサーコストが増大
- 代替（IMU+BLE）で機能的に代替可能

### 2.2 個人適応設計

#### 採択案：4層適応アーキテクチャ（固定ベース + 更新可能な上位レイヤー）

```
Layer 4: 周期的LoRA（研究フェーズ）  ← 週次バッチ更新
Layer 3: 行動ポリシーモジュール       ← セッション内リアルタイム更新
Layer 2: エピソード記憶RAG           ← セッション後に更新
Layer 1: ユーザプロファイル注入       ← 都度更新可能
Layer 0: 固定ベースLLM (14B Q4)      ← 事業期間中不変
```

**Layer 0が固定である理由：** 医療環境で使用するシステムにおいて、基盤モデルの安全フィルタ・一般能力が予測不可能に変化するリスクは許容できない。基盤は固定し、その上に安全に制御できる適応層を積む。

**Layer 4がオンラインではなく周期バッチである理由：**
- オンラインLoRA（推論中のリアルタイム勾配計算）は推論コストを2〜4倍に増加させ、14B Q4では実質的に不可能
- 単一会話でのドリフトリスク（数件の誤ったフィードバックで応答が変質する）
- 臨床倫理上、学習データとしての会話使用に事前同意と事後確認が必要→バッチ管理が必然

**破滅的忘却対策（Layer 4）：**
- リプレイバッファ：学習データの10〜20%に汎用日本語対話データを混入
- 適応前後のベンチマーク比較（汎用対話品質指標）を自動チェック
- 閾値を下回った場合に前バージョンのLoRAアダプタへ自動ロールバック
- LoRAアダプタはbase weightsに触れないため、base capabilityは本質的に保護される

#### 棄却案：ソフトプロンプト（Prefix Tuning）
**棄却理由：**
- ベースモデルのバージョンアップ時に全ソフトプロンプトの再訓練が必要
- 医療施設での長期サポートを考えると保守コストが高すぎる
- LoRAに比べて解釈可能性が低く、安全フィルタとの相互作用が不透明

#### 棄却案：MemGPT方式（LLM自身がメモリ管理）
**棄却理由：**
- LLMが自身のメモリを管理するため追加の推論コスト（memory reads/writes）が発生
- 14B Q4モデルでのemergent memory management能力は不確実
- 施設環境のように専門知識ドメインが固定されている場合はシンプルなRAGで十分

### 2.3 デバイスハードウェア設計

#### 採択案：RP2040（常時ON MCU）+ RPi Zero 2W（メインSBC）

**比較評価：**

| 特性 | RP2040+Zero2W | ESP32-S3単体 | RPi 4B単体 |
|---|:---:|:---:|:---:|
| BOM（単品） | ¥8,850 | ¥6,000 | ¥16,000 |
| WiFi音声ストリーミング | ◎ | △（RAM制限） | ◎ |
| 常時ON電力 | ◎（~30mW） | ◎（~25mW） | △（~3W） |
| Linux環境 | ◎（OTA/更新容易） | ✗ | ◎ |
| オフラインフォールバック | △（キャッシュのみ） | ✗ | ○（Whisper-tiny可） |
| 量産BOM（500台） | ¥6,500〜 | ¥4,000〜 | ¥12,000〜 |
| 開発生態系 | ◎ | ○ | ◎ |

**RP2040+Zero 2Wを選んだ理由：**
RP2040（MicroPython/C++）が常時ON処理（VAD・IMU・LED・触覚）を~30mWで担当し、Zero 2W（Linux）は必要時のみフル起動してOpus圧縮・WiFiストリーミング・ハイレベル制御を担当する分業が、コスト・電力・開発容易性・保守性のバランスで最良。

**ESP32-S3が惜しかった点と棄却理由：**
コストは最良だが、8MB PSRAMでOpus圧縮+WiFi送信を同時安定実行する余裕が薄い（実証なし）。v1フィールドで問題が起きた場合の原因特定が困難。v2（量産）段階でのコスト最適化候補として保留。

### 2.4 計算分散設計

#### 採択案：施設内オンプレミスサーバ（ASR/LLM/TTS）

**採択理由：**
- 医療・介護施設での患者会話データのクラウド送信は個人情報保護法・医療情報ガイドラインとの整合が困難
- 施設LAN内の通信はRTT<15msで安定しており、ユーザ体験を損なわない
- Mac mini M4 Pro（約15万円）は施設IT部門が管理できる製品カテゴリ

**「完全ローカル（全処理をデバイス上）」を棄却した理由：**
- 14B Q4モデルはデバイスサイズ（BOM ¥10,000以下）では実行不可能
- Whisper-large-v3やStyle-Bert-VITS2もデバイス上での実行は計算量的に不可
- 仮にモデルを7B以下に圧縮しても、温度・騒音変動の多い施設環境での音声品質を確保するには十分な計算資源が必要

**ハイブリッドクラウドモデル（将来フェーズ）：**
- 在宅展開向け。プライバシー設定に応じてエンドツーエンド暗号化でクラウド推論を使用
- ASRをwhisper.cpp + fastAPIのサーバレス関数で、LLMをGroq/TogetherAI（低遅延APIサービス）で補完
- 事業期間外の展開として計画

**ネットワーク断絶時のフォールバック（実装済み計画）：**
- デバイスSDカード（8GB）に25種の定型応答TTS音声をキャッシュ
- Zero 2W上でローカル判定：サーバ応答なし→キャッシュ応答を再生
- 「現在確認中です、少々お待ちください」等の橋渡し応答でユーザの不安を軽減

## 3. レイテンシ詳細設計

### 目標と現状のギャップ

| 段階 | 現プロトタイプ | 目標（v2） | 最適化手法 |
|---|:---:|:---:|---|
| VAD終端検出 | 500ms | 300ms | 窓サイズ縮小 + 終端感度調整 |
| WiFi送信 | 30ms | 20ms | Opusビットレート最適化 |
| ASR (5秒発話) | 600ms (Whisper large) | 300ms (distil-Whisper) | モデル変更 |
| RAG検索 | 100ms | 30ms | FAISSフラットインデックス + 事前ウォームアップ |
| LLM prefill | 1500ms | 800ms | コンテキスト削減（600トークン上限） |
| LLM 第1文生成 | 800ms | 400ms | 短文生成ターゲット（20トークン以内） |
| TTS 第1文 | 700ms | 400ms | ストリーミング + センテンスセグメンテーション |
| WiFi返送+バッファ | 150ms | 80ms | JitterBuffer削減 |
| **合計（RTT）** | **4.4秒** | **2.3秒** | |

### 投機的プリフィルの設計

VAD窓が「発話終端に近づいている」と判断した時点（終端確定500ms前）でRAG検索を開始し、ASR完了後にLLMへ即座に投入。これによりRAG検索時間30msをレイテンシ経路から除外できる。

## 4. 技術的リスクと軽減策

| リスク | 確率 | 影響 | 軽減策 |
|---|:---:|:---:|---|
| RPi Zero 2Wのオーディオ品質不足 | 中 | 中 | ReSpeaker 2-Mic HATのハードウェアエコーキャンセリング使用; 問題時はRP4への格上げ |
| distil-Whisperの日本語精度 | 中 | 高 | 施設特有語彙（薬名・人名）をWhisperの初期プロンプトで誘導; 評価後にWhisper-mediumへ戻す判断も保留 |
| 施設WiFiの不安定性 | 高 | 中 | フォールバック機構 + JitterBuffer; 施設導入パッケージにAP設置を含める |
| LoRAバッチの倫理承認遅延 | 中 | 低 | Layer 4をオプション機能として設計; Layer 1-3のみで主要価値は提供できる |
| 14B Q4の日本語医療用語応答品質 | 中 | 高 | 施設ドメイン知識をRAGに完全収録し、LLMは推論・言語化のみに使用; ドメインRAGで質を担保 |

## 5. 看護ログ生成の設計と安全設計

### パイプライン

```
会話テキスト（セッション後）
  ↓
NER（固有表現認識: GiNZA / SudachiPy）
  → 人名・薬品名・症状語・時間表現を抽出
  ↓
イベント分類器（fine-tuned BERT small, 医療ドメイン）
  → カテゴリ: [不調訴え / 活動報告 / 服薬確認 / 不安表明 / その他]
  ↓
構造化JSONレコード生成
  ↓
スタッフダッシュボード（FastAPI + シンプルWeb UI）
  → 看護師が確認・承認・修正
  ↓
（オプション）施設の記録システムへの手動転記
```

### 安全設計の原則

1. **自動生成ログは参考情報のみ**：電子カルテへの直接書込は行わない。看護師の確認・承認を必須とし、確認前のレコードには「未確認」フラグを表示
2. **責任の明確化**：システムは情報の「整理ツール」であり、臨床判断の根拠としての使用を禁止する旨を利用規約・施設合意書に明記
3. **誤認識の表示**：NERの信頼スコアが閾値以下のレコードには「要確認」マーカーを表示
4. **患者の訂正権**：患者がSuzume-chanに「さっきの話は記録しないで」と言うと、当該セッションの看護ログをドラフトから削除する仕組みを実装
